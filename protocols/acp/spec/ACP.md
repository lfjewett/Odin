# Agent Chart Protocol (ACP)

- Spec ID: `ACP-0.1.0`
- Status: Draft (Authoritative in this repository)
- Last updated: 2026-03-01

## 1) Purpose

ACP defines a uniform contract for chart data agents used by Odin.

ACP standardizes:
- Historical backfill (HTTP REST)
- Real-time streaming (WebSocket)
- Subscription lifecycle
- Error handling
- Heartbeats and liveness semantics
- Rendering hints for UI overlays
- Record-level dedup and replay behavior

This repository is the source of truth for ACP behavior and schema.

## 2) Scope and Non-Goals

### In Scope
- Single-symbol chart sessions
- One subscription maps to one agent + one symbol + one interval + parameter set
- Stateless agent behavior per subscription
- At-least-once delivery
- JSON payloads

### Non-Goals (for `ACP-0.1.0`)
- Multi-symbol subscriptions
- Agent discovery/registry protocol
- Authentication/authorization (local-only deployment)
- Binary serialization (e.g., Protobuf/MessagePack)

## 3) System Roles

### Backend (Router)
- Initiates and manages subscriptions to agents
- Calls each agent REST history endpoint for backfill
- Opens and manages WebSocket connections to each agent for live data
- Merges normalized streams and forwards to UI
- Applies deduplication and gap handling

### Agent
- Exposes mandatory HTTP backfill API
- Exposes mandatory WebSocket live stream API
- Is stateless per subscription
- Emits records in supported ACP schemas
- Emits heartbeat and errors on the stream

### UI
- Consumes merged backend stream
- Does not compute indicators
- Uses overlay hints only for presentation

## 4) Transport Model

## 4) Transport Model

## 4.1 Mandatory Interfaces
Each ACP agent MUST provide:
1. REST API for agent metadata (`/metadata`)
2. REST API for historical data (`/history`)
3. WebSocket API for live data

Live data MUST NOT be polled over REST.

## 4.2 Connection Direction
- Backend connects to agents.
- Backend interrogates agent metadata via `/metadata` endpoint before subscribing.
- Backend requests history via agent REST endpoints.
- Backend connects to agent WebSocket and sends subscription lifecycle messages.

No agent discovery protocol is defined in `ACP-0.1.0`.

## 5) Data Schemas

Supported record schemas:
- `ohlc`
- `line`
- `event`
- `band`
- `histogram`
- `forecast`

All records MUST include:
- `id` (string; stable for dedup)
- `ts` (UTC ISO-8601 timestamp)

See the `schemas/` directory for normative JSON Schemas.

## 6) Lifecycle

## 6.0 Canonical Intervals

For `ACP-0.1.0`, valid interval values are:
- `1m`
- `5m`
- `15m`
- `30m`
- `1h`
- `4h`
- `1d`

`interval` in all ACP requests and messages MUST use one of these values.

## 6.1 Subscription Identity
`subscription_id` is generated by backend and unique in backend scope.

A subscription is uniquely defined by:
- `agent_id`
- `symbol`
- `interval`
- `params`

Agents MUST treat each subscription independently and MUST NOT require shared state across subscriptions.

## 6.2 WebSocket Messages (Backend -> Agent)

### Subscribe
```json
{
  "type": "subscribe",
  "spec_version": "ACP-0.1.0",
  "subscription_id": "sub-123",
  "agent_id": "sma_20",
  "symbol": "SPY",
  "interval": "1m",
  "params": {
    "length": 20
  }
}
```

### Unsubscribe
```json
{
  "type": "unsubscribe",
  "spec_version": "ACP-0.1.0",
  "subscription_id": "sub-123"
}
```

### Reconfigure (optional support)
If supported by agent, behavior MUST be equivalent to:
1. `unsubscribe(old)`
2. `subscribe(new)`

```json
{
  "type": "reconfigure",
  "spec_version": "ACP-0.1.0",
  "subscription_id": "sub-123",
  "symbol": "SPY",
  "interval": "5m",
  "new_params": {
    "length": 50
  }
}
```

If unsupported, agent MUST return `UNSUPPORTED_OPERATION`.

## 7) Historical Backfill (REST)

Required endpoint:

`GET /history?symbol={symbol}&from={iso8601}&to={iso8601}&interval={interval}`

Response:
```json
{
  "spec_version": "ACP-0.1.0",
  "agent_id": "price_agent",
  "schema": "ohlc",
  "symbol": "SPY",
  "interval": "1m",
  "data": [
    {
      "id": "bar-18392",
      "seq": 18392,
      "ts": "2026-02-01T14:32:00Z",
      "open": 492.1,
      "high": 492.4,
      "low": 491.9,
      "close": 492.3,
      "volume": 19342
    }
  ]
}
```

Rules:
- `id` MUST be stable for deduplication.
- If present, `seq` MUST be monotonic within a stream.
- Missing `seq` values indicate a gap.
- `from` is inclusive, `to` is exclusive.
- Records MUST be ordered ascending by `ts`.
- `interval` MUST be one of the canonical interval values in section 6.0.
- Backfill responses for `ohlc` MUST return finalized bars only (`bar_state = final`).

## 8) Delivery Semantics and Dedup

ACP uses at-least-once delivery.

Deduplication keys:
- Non-`ohlc` schemas: `(agent_id, id)`
- `ohlc` partial/final stream: `(agent_id, id, rev)`

Backend behavior:
- Duplicate non-`ohlc` key => ignore duplicate record
- Duplicate `ohlc` key `(agent_id, id, rev)` => ignore duplicate record
- For `ohlc`, treat higher `rev` for the same `(agent_id, id)` as an upsert of the same candle
- Gap detected (`seq` discontinuity when `seq` is used) => trigger backfill

## 9) Liveness and Health

ACP heartbeat defines stream liveness only.

Recommendations:
- Agent sends heartbeat every 5-15s per active subscription.
- Backend marks subscription stale if no heartbeat for 3x interval.
- Agent-local health checks are implementation-specific and out of protocol scope.

## 10) Agent Metadata and Configuration (REST)

All ACP agents MUST provide a metadata endpoint for configuration discovery and UI rendering hints.

Required endpoint:

`GET /metadata`

Response:
```json
{
  "spec_version": "ACP-0.1.0",
  "agent_id": "sma_20",
  "agent_name": "Simple Moving Average",
  "agent_version": "1.0.0",
  "description": "Computes simple moving average over price data",
  "config_schema": {
    "length": {
      "type": "integer",
      "description": "Number of periods for moving average",
      "default": 20,
      "min": 1,
      "max": 200,
      "required": true
    }
  },
  "output_schema": "line",
  "overlay": {
    "kind": "line",
    "panel": "price",
    "y_range": null,
    "color": "#3b82f6",
    "line_width": 2,
    "legend": "SMA(20)"
  }
}
```

### Metadata Fields

**Required fields:**
- `spec_version` (string): ACP version the agent implements
- `agent_id` (string): Unique identifier for this agent type
- `agent_name` (string): Human-readable agent name
- `agent_version` (string): Agent implementation version
- `description` (string): Brief description of agent functionality
- `config_schema` (object): Configuration parameters the agent requires
- `output_schema` (string): One of: `ohlc`, `line`, `event`, `band`, `histogram`, `forecast`
- `overlay` (object): UI rendering hints

### Config Schema Format

Each configuration parameter in `config_schema` MUST include:
- `type`: One of `string`, `integer`, `number`, `boolean`
- `description`: Human-readable explanation
- `required`: Whether the parameter is mandatory
- `default` (optional): Default value if not provided

Type-specific optional fields:
- For `integer`, `number`: `min`, `max`
- For `string`: `pattern` (regex), `enum` (allowed values)

### Overlay Hints

The `overlay` object guides UI rendering:

**Required fields:**
- `kind`: Matches `output_schema` - one of: `ohlc`, `line`, `event`, `band`, `histogram`, `forecast`
- `panel`: `price` (overlay on price chart) or `separate` (dedicated panel)

**Optional fields:**
- `color`: Hex color code (e.g., `#3b82f6`)
- `legend`: Display label for the indicator
- `y_range`: `[min, max]` or `null` for auto-scale
- `line_width`: Stroke width for line-based overlays
- `style`: Rendering style hints (agent-specific)

### Backend Behavior

Backend MUST:
1. Call `/metadata` before establishing subscriptions to validate configuration
2. Store and forward overlay hints to UI
3. Validate user-provided configuration against `config_schema`

### UI Behavior

UI MUST:
1. Interrogate agent `/metadata` when user adds new agent
2. Display configuration form based on `config_schema`
3. Apply overlay hints for rendering agent output

## 11) Price Agent Norms

Price agent is the canonical `ohlc` source.

Price agent MUST:
- Emit real-time partial updates for the currently open bar
- Emit a final update when a bar closes
- Set `bar_state` to `partial` for in-progress updates and `final` for closed bar updates
- Provide monotonic `rev` per `id` where each new update for the same bar increments `rev`

Price agent SHOULD:
- Target sub-second to 1-second partial update cadence depending on source feed availability
- Assign monotonic `seq`
- Persist history for replay and backfill

`ohlc` identity and revision rules:
- `id` identifies the logical candle and remains stable from first partial through final update
- `rev` is monotonic per `id`
- `bar_state = final` is terminal for the bar and MUST NOT be mutated afterward

Indicator and analysis agents SHOULD:
- Depend on canonical price stream semantics
- Handle missing sequence windows deterministically
- Recompute deterministically for replay consistency

## 12) Error Codes

Structured error codes for `ACP-0.1.0`:
- `INVALID_REQUEST`
- `INVALID_SYMBOL`
- `INVALID_INTERVAL`
- `INVALID_PARAMS`
- `UNSUPPORTED_OPERATION`
- `SUBSCRIPTION_NOT_FOUND`
- `AGENT_OVERLOADED`
- `BACKFILL_TIMEOUT`
- `INTERNAL_ERROR`

Error payload fields:
- `code` (required)
- `message` (required)
- `retryable` (required)
- `details` (optional object)

## 13) Versioning

- Current protocol version: `ACP-0.1.0`
- Every ACP message MUST include `spec_version`

Compatibility policy:
- Patch (`0.1.x`): clarifications / non-breaking schema additions
- Minor (`0.x.0` where `x` changes in pre-1.0): may include breaking changes in draft era
- Major (`1.0.0+`): strict semantic versioning with explicit breaking-change process

## 14) Compliance Requirements

An implementation is ACP-0.1.0 compliant if it:
1. Implements REST `/history`, `/metadata`, and WebSocket live stream
2. Supports `subscribe` and `unsubscribe`
3. Emits `data`, `heartbeat`, and structured `error`
4. Provides complete agent metadata via `/metadata` including config schema and overlay hints
5. Uses JSON payloads with required fields
6. Uses at-least-once delivery and dedup semantics from section 8
7. Operates statelessly per subscription
8. Restricts subscriptions to exactly one symbol per subscription
9. Uses only canonical interval values from section 6.0
10. For `ohlc`, supports partial/final streaming with `bar_state` and monotonic `rev`
